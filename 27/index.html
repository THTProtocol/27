Create

Player A creates a match, selects game type and escrow amount. A P2SH address is generated client-side and the escrow TX is broadcast to the Kaspa network.

P2SH script: OP_IF [player_a_pubkey] OP_CHECKSIG OP_ELSE [timeout] OP_CHECKLOCKTIMEVERIFY OP_DROP [player_b_pubkey] OP_CHECKSIG OP_ENDIF. Escrow UTXO is locked to this script. Firebase doc stores match metadata, escrow txid, and P2SH address.

Accept

Player B joins by matching the escrow amount. Both deposits are now locked in the P2SH address until the game resolves.

Player B sends their escrow to the same P2SH address. Firebase match doc is updated with player B's address and the acceptance txid. Game clock starts upon acceptance confirmation (1 block).

Play

Players compete in real-time. All moves are synced via Firebase Realtime Database and validated on both clients.

Game state is stored at /matches/{id}/board in Firebase RTDB. Each move is validated by chess.js (or equivalent engine) on both clients. Time controls are enforced client-side with Firebase server timestamps for sync. Disconnect detection: 30s grace period, then forfeit.

Resolve

The oracle attests to the game result. The signed attestation is published to Firebase and can be independently verified.

Oracle signs: SHA256(matchId + winner + finalState). The attestation uses a Schnorr signature (secp256k1). For ZK path: a Groth16 proof is generated off-chain and verified on-chain via KIP-16. Settlement is possible with either oracle attestation or ZK proof.

Settle

The winner's browser constructs and broadcasts the settlement TX, sending the escrowed funds (minus protocol fee) to the winner's address.

Settlement TX spends the P2SH UTXO using the oracle's signature as the witness. Protocol fee (2%) is sent to the HTP fee address. Remaining funds go to the winner. TX is broadcast via the connected wRPC node. Confirmation is tracked and displayed in the match history.

Oracle Network

Decentralized resolution infrastructure: ZK proofs, bonded oracles, and miner attestation.

ZK Verification Oracle: Active

Zero-Knowledge Verification

Game results are verified using a SHA-256 commit-reveal scheme with an optional Groth16 ZK proof path (KIP-16). The prover commits to the game state hash before the match begins, then reveals the preimage upon completion.

Network

Testnet 12 (TN12). High Table Protocol uses the Kaspa testnet for development and testing. Switch to mainnet via URL parameter: ?net=mainnet

BlockDAG

Kaspa uses a BlockDAG (directed acyclic graph) instead of a linear blockchain. Multiple blocks can be mined in parallel, enabling high throughput and fast confirmations.

PHANTOM/GhostDAG

The GhostDAG consensus protocol orders blocks in the DAG, assigning blue scores to honest blocks and maintaining security under high block rates (currently 10 BPS).
