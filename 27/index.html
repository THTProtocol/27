Match Lifecycle

Create
Lock KAS in P2SH escrow contract. Generates invite link for opponent.
P2SH address derived from match parameters. UTXO locked with 2-of-2 multisig until oracle resolution.
Accept
Opponent matches escrow. Both funds locked in contract.
Second UTXO added to escrow. Total pot = 2x escrow minus protocol fee.
Play
Game begins. Moves synced in real-time via Firebase. Clocks enforced.
Move validation on client and server. Clock managed by htp-chess-sync with Firebase serverTimestamp.
Resolve
Oracle verifies result. ZK proof submitted or bonded oracle attests.
Oracle daemon monitors match completion. Generates attestation or ZK proof for on-chain verification.
Settle
Winner receives pot. Transaction broadcast to Kaspa network.
Auto-payout engine constructs settlement TX. Winner receives (2x escrow - fee). Loser escrow released to winner address.

Resolution Pipeline

1. SHA-256 commit scheme: hash(result || salt) posted pre-game
2. Game completes: move log + result submitted
3. KIP-16 Groth16 circuit verifies game integrity
4. ZK proof submitted on-chain for settlement

Terms of Service

High Table Protocol is experimental software operating on the Kaspa testnet (TN12). All matches and predictions involve testnet KAS with no real-world value. Use at your own risk. The protocol is provided as-is with no warranty. Smart contract escrow is enforced on-chain but oracle resolution depends on network participants. Do not use real funds or mainnet KAS until the protocol has been audited and deployed to mainnet.
